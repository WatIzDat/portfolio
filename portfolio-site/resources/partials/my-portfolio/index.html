My Portfolio

This is the site you are currently on right now. This article will detail the steps that went into making this project, and how I would redo it if I started over. This was the first ever piece of software I wrote in Clojure/ClojureScript, so it took a lot of trial and error and I certainly made a lot of mistakes, but I think it turned out fairly well.

First, let me give some backstory. Before starting this project, I already knew I wanted to make a portfolio. However, I had been working in Java at the time, which was getting more and more frustrating the more I worked with it. Furthermore, I was using React, which was fine, but was still a bit annoying to work with. That's how I came across Clojure. I had heard many good things about Clojure, and being on the JVM and all, I thought it was perfect for a Java dev. The only roadblock was, of course, that Clojure is a functional language, so it took a while to get used to for my OOP-conditioned brain.

I started out by grinding 4clojure problems, which is essentially a LeetCode-type platform for Clojure. It helped a lot, and as I became more familiar with the standard library, I was enjoying the language more and more. So after making some test projects, I finally decided to embark on this project.

The Architecture

This project is separated into three parts: the portfolio-api, the portfolio-manager, and the portfolio-site.

The portfolio-api is the backend for the portfolio-manager. It interacts with a database and ensures that all articles are securely stored. The portfolio-manager is the internal management site that I wrote. It uses a WYSIWYG editor to enable me to write articles easily. Lastly, the portfolio-site is the site that you are on right now. It takes articles written with the portfolio-manager and makes them pretty for viewing.

The Stack

As stated previously, this project uses 100% Clojure and ClojureScript. As the build tool I chose Leiningen. For the portfolio-api, I used Ring as an HTTP abstraction and Reitit for routing. Malli was used for validation, Mount was used for state, and HoneySQL was used to interact with a Postgres database. For the portfolio-manager, I used Reframe, and finally, for the portfolio-site, I simply used stasis to generate a static site. Both of the latter two use TailwindCSS for styling.

The API

The portfolio-api is nothing special. It has a few CRUD endpoints for articles, and that's about it. I did thoroughly enjoy working with the stack I chose, since it was a small project and no choice would have been too detrimental. The biggest roadblock I had is that, because my brain is so conditioned to working with frameworks as opposed to libraries (I work in .NET/C# mainly), the thought of having to self-assemble different libraries together irked me, but I quickly realized that having so much control over your own stack is a great benefit.

The Manager

I had significantly more problems with the portfolio-manager than the API. Not only am I not a frontend person, I also just had no idea what I was doing with Reframe. If I had to redo it, I would have used Rum rather than Reframe, since I don't like the complex indirection Reframe uses to manage state. I would much rather just directly talk to a big global atom for state, or maybe through a function. But I am a naive beginner, so take my opinion with a bucket of salt.

Another problem I had was the WYSIWYG editor. At first, I used Trix, but quickly realized that it has minimal features (only one header, for example), so then I decided to switch to Quill. There's two main problems with Quill: it's not built for React, and it uses the Quill Delta format which doesn't have implementations anywhere else. I haven't bothered to switch yet, but I will do some research later and switch to an editor that will better fit my use case.

The Site

And last but not least, the site that you are currently reading this information from. At first, I made the mistake of not considering my use case more, and settled on Rum. However, I quickly realized that for such a website with zero interactivity, a static site would be so much simpler. It was difficult to choose at the start, since it didn't seem like any static site generator fit my use case particularly well. This is because, as I mentioned before, I am using the Quill Delta format, and most static site generators use Markdown or AsciiDoc. Luckily, I found stasis, which gives the developer full control of their paradigm due to its simplicity, allowing it to work perfectly for me.

Reflection

Overall, I enjoyed my experience working with Clojure and the Clojure ecosystem. This wasn't a big project by any means, so I might be making these comparisons too eagerly, but I believe I am certainly enjoying this stack better than C# and Next.js, which is what I mainly used in the past. I don't have too many regrets about my decisions when writing this project, except for the portfolio-manager, which I would switch to Rum and a different editor if I rewrote it. If you are considering learning Clojure, please go for it. Trust me, the parenthesis are hardly even an issue (it took me until the very end of the article for me to mention it), and it's a beautiful language. I will warn you though: it's tough to go back once you've fallen for it.
